# Data Card — Claims-Risk Orchestrator

All training data is synthetic. Generated by `src/generate_data.py`
with a fixed seed (42) for reproducibility. The shape of the data is
modeled on three real public datasets you would use in production —
this card explains the mapping.

## Synthetic generators ↔ real-world analogues

| Generated file | Real analogue | What it represents |
|----------------|---------------|--------------------|
| `data/claims.csv` | CMS DE-SynPUF (Synthetic PUF) | Insurance claim cost / utilization |
| `data/labs.csv` | NHANES lab panel data | Patient-level lab measurements |
| `data/notes.csv` | MIMIC-IV discharge notes | Free-text clinical narratives |

Swapping in the real datasets requires no code changes to the agents,
tools, or orchestrator — only the loaders and the schema mapping.

---

## `data/claims.csv`

### Shape
~2,000 rows × 7 columns

### Columns
| Column | Type | Description |
|--------|------|-------------|
| `patient_id` | str | Synthetic ID (`P00000`–`P01999`) |
| `cost_usd` | float | Lognormal(mean=8.5, sigma=1.0); typical range $1k–$50k |
| `procedure_count` | int | Uniform integer 1–7 |
| `los_days` | int | Length of stay; uniform integer 0–13 |
| `drg_code` | str | One of {470, 291, 292, 871} |
| `age` | int | Uniform integer 45–89 |
| `is_anomaly` | int | Target. 1 if `cost > mean+std AND procedure_count >= 5`, plus 5% random label noise |

### Generation method
- `np.random.default_rng(42)` for full reproducibility
- Cost distribution: lognormal — mimics real medical billing where most
  claims are low and a long tail is high
- 5% noise injected on the label to simulate genuine ambiguity / clerical
  miscoding in real claim labels
- Class imbalance: ~10–15% positive rate, in line with industry
  fraud-flag prevalence (1–10% depending on payer)

### Drift considerations
- The `drg_code` distribution is fixed; in production this is highly
  payer- and specialty-dependent
- The cost distribution does not include payer-specific contracted rates
- No seasonality (real claims peak in Q4 due to deductible reset)

## `data/labs.csv`

### Shape
~1,800 rows × 6 columns (10% of the base 2,000 are dropped to simulate
patients with no submitted lab panel)

### Columns
| Column | Type | Description |
|--------|------|-------------|
| `patient_id` | str | Joins to `claims.csv` |
| `a1c` | float | Hemoglobin A1C %, Normal(6.5, 1.5) |
| `ldl` | float | LDL cholesterol mg/dL, Normal(130, 35) |
| `egfr` | float | Estimated GFR, Normal(70, 20) |
| `troponin` | float | Cardiac troponin ng/mL, Exponential(scale=0.05) |
| `at_risk` | int | Target. 1 if any panel breaches its clinical threshold |

### Clinical thresholds used for the `at_risk` label
- A1C > 8.5 (uncontrolled diabetes)
- LDL > 190 (severe hypercholesterolemia)
- eGFR < 45 (CKD stage 3b+)
- troponin > 0.4 (cardiac injury)

### Drift considerations
- Distributions are independent normals. Real labs are correlated
  (e.g. eGFR and A1C in diabetic kidney disease)
- No timestamp / temporal ordering; real lab feeds are time-series
- Missingness is uniform random; real missingness is informative
  (sicker patients have more labs)

## `data/notes.csv`

### Shape
~2,000 rows × 3 columns

### Columns
| Column | Type | Description |
|--------|------|-------------|
| `patient_id` | str | Joins to `claims.csv` |
| `discharge_note` | str | One of 9 templated sentences across 3 severity buckets |
| `true_severity` | str | `low` / `med` / `high` — the bucket used to sample the note |

### Templates
- Low (60% of rows): "Patient stable post-operative. Adherent to medications. Family supportive." (and 2 similar)
- Med (25% of rows): "Patient with prior admission 18 days ago. Concern for medication non-adherence." (and 2 similar)
- High (15% of rows): "Workup completed. Findings inconsistent with billed procedures. Awaiting review." (and 2 similar)

### Why templates instead of LLM-generated text
Reproducibility. Template selection is seeded; an LLM-generated corpus
would not be bit-reproducible across runs and would inflate the project's
external dependencies.

### Drift considerations
- Real discharge notes vary in length from 200 to 5,000 tokens; these
  templates are 10–30 tokens
- Real notes contain identifiers, abbreviations, mis-spellings, and
  clinical jargon that the templates omit entirely
- The `true_severity` ground truth is the *intent* of the template,
  not a clinically validated label

## Privacy / ethics statement

- All data is synthetic. There is no PHI, no real patients, no real
  claims, and no consent process to administer
- `tests/test_secrets.py` guards against committed Gemini API keys
- A real deployment would require a HIPAA Business Associate Agreement
  with the LLM provider, plus IRB review for any retrospective study
  using the outputs
- The architecture does not compute or use protected-class attributes
  (race, ethnicity, gender beyond what's incidentally present in age);
  a production version would need an explicit fairness audit before
  going live
