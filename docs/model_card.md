# Model Card — Claims-Risk Orchestrator

Two ML models ship in this project. Both are trained on synthetic data
intended for architecture demonstration, not deployment. Numbers below
are reproducible from `python -m src.train_claims_rf` and
`python -m src.train_labs_nn`.

---

## Model 1 — Claims Anomaly Random Forest

### Model details
- **Architecture:** scikit-learn `RandomForestClassifier`, 500 trees,
  `max_depth=None`, `min_samples_leaf=2`, `min_samples_split=5`,
  `max_features='sqrt'`, `class_weight='balanced'`.
- **Preprocessing:** `StandardScaler` on numeric features, `OneHotEncoder`
  on `drg_code`. Bundled into a single `Pipeline` so inference is one call.
- **Hyperparameter selection:** Optuna TPE sampler with 5-fold CV,
  ~30 trials. See `src/tune_claims_rf.py`.
- **Version / artifact:** `models/claims_rf.joblib`,
  metrics in `models/claims_rf_metrics.json`,
  feature importances in `models/claims_rf_feature_importance.json`.
- **Tracking:** every training + tuning run is logged to MLflow under
  experiments `claims-risk-rf` and `claims-risk-rf-sweep`.

### Intended use
- Score whether a single claim is anomalous given numeric and categorical
  features, returning a probability in [0, 1] with SHAP-attributed top
  features for explainability.
- Consumed by the `ClaimsAgent` LlmAgent via the `predict_claims_anomaly`
  FunctionTool. The agent passes the score to the FusionAgent for
  context-aware weighting against the labs and notes signals.

### Out of scope
- Real PHI / production claims. The model has never seen real data.
- Any patient-level decisioning. The pipeline outputs a routing
  recommendation, not a diagnosis or coverage determination.

### Training data
See `docs/data_card.md`. ~2,000 synthetic claim rows generated by
`src/generate_data.py` with a deterministic seed.

### Evaluation
- 80/20 stratified train/test split (seed=42)
- Stratified 5-fold CV AUC on the training portion to estimate variance
- Held-out AUC reported as the primary metric
- Calibration: Brier score + 10-bin reliability curve (see
  `claims_rf_metrics.json:calibration`)
- Per-class precision / recall / f1 in the same metrics file

### Known limitations
- Trained on ~2k rows of synthetic data. Real claims distributions are
  long-tailed; expect substantial drift if applied to production data.
- The class-imbalance handling is `class_weight='balanced'`, not
  oversampling. SMOTE / focal loss would likely improve recall on the
  minority class.
- No demographic / fairness audit. The synthetic generator emits an
  `age` feature but no protected-class attributes; a real deployment
  would need bias evaluation against the FTC Equal Credit Opportunity
  Act analogues (or HIPAA Privacy Rule for healthcare).

### What would make this production-ready
- Real CMS DE-SynPUF + outcome labels for retrospective evaluation
- Calibration via Platt scaling or isotonic regression on a held-out set
- Drift monitoring (input distribution + score distribution + outcome lag)
- Reject-option classification for the low-confidence band
- Model registry with promotion gates (MLflow Model Registry or Vertex AI
  Model Registry)

## Model 2 — Labs Risk DNN (TensorFlow)

### Model details
- **Architecture:** Keras Sequential — Dense(32, ReLU) → Dropout(0.2)
  → Dense(16, ReLU) → Dropout(0.2) → Dense(1, sigmoid).
- **Preprocessing:** `StandardScaler` fit on training labs; saved
  alongside the model (`models/labs_scaler.joblib`).
- **Optimizer:** Adam, lr=1e-3, binary cross-entropy, 20 epochs,
  batch size 64, validation split 0.1, seed=42.
- **Version / artifact:** `models/labs_nn.keras`,
  `models/labs_scaler.joblib`,
  metrics in `models/labs_nn_metrics.json`.
- **Tracking:** MLflow experiment `claims-risk-labs-nn`.

### Intended use
- Score lab-panel risk from four numeric panels: A1C, LDL, eGFR,
  troponin. Returns a probability in [0, 1].
- Consumed by `LabsAgent` via the `score_labs_risk` FunctionTool.
- Imputation: missing values are mean-filled at inference time and the
  agent's confidence is down-weighted by 30%.

### Out of scope
- Anything beyond the four-panel signal. A real lab integration would
  use the full LOINC code space and longitudinal trends.

### Training data
~1,800 rows of synthetic labs (10% are dropped from the ~2,000 base to
simulate the real-world "missing panel" case). See `docs/data_card.md`.

### Evaluation
- 80/20 stratified split, seed=42
- Held-out AUC and accuracy as primary metrics
- Brier score for calibration
- Final train AUC and final validation AUC tracked from training history

### Known limitations
- Same synthetic-data caveat as the RF.
- Imputation by mean is the simplest possible strategy; in production
  you would use multiple imputation, mask-aware models, or train
  separate models per missingness pattern.
- The architecture is intentionally tiny — chosen for "runs in 30s on a
  laptop CPU," not for predictive ceiling.

## Combined system risk surface

When the two ML models feed into the LLM-driven Fusion + Review +
Action layer, several failure modes compound:

- **Model drift in production** would silently change the input
  distribution that the FusionAgent's prompt assumes. Mitigation:
  monitor `score` and `confidence` distributions per agent and alert
  on shifts.
- **LLM nondeterminism** in Fusion / Reviewer / Action means decisions
  are not reproducible across runs. Mitigation: the `eval/run_eval.py`
  harness reports `decision_agreement_rate` to surface this.
- **Tool-output trust:** the Action layer trusts the FusionAgent's
  fields verbatim. A prompt-injection in a discharge note could in
  principle steer the recommendation. Mitigation: the ReviewerAgent
  rule R1 (confidence ≥ 0.65) acts as a guardrail; a real deployment
  would add output-schema validation and constitutional checks.

## Reproducibility

```bash
python -m src.generate_data --n 2000 --out data
python -m src.train_claims_rf
python -m src.train_labs_nn
mlflow ui --backend-store-uri ./mlruns      # browse runs
```

All seeds (RF, NN, train/test split, CV) are fixed at 42. Re-running
the pipeline on the same machine should produce bit-identical
predictions; minor floating-point differences may appear across CPU
architectures.
